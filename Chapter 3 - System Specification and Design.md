### 3.1 Introduction

This chapter presents the system specification and design of the Smart Attendance System proposed for Ghana Communication Technology University (GCTU). Building upon the research gaps identified in Chapter Two — specifically the absence of simultaneous QR code dynamism, GPS-based physical verification, and immutable record storage in a single integrated system — this chapter translates those research findings into a concrete, implementable design. The chapter begins by specifying the functional and non-functional requirements that the system must satisfy, derived from a careful analysis of the institutional context and the technical shortcomings of existing systems reviewed in the literature. It then presents the overall system architecture, describing how the major components interact. The chapter proceeds to detail the use case design, data flow diagrams at multiple levels of abstraction, process flowcharts, the entity relationship diagram governing the database, and the interface design of the student mobile application and the lecturer web dashboard. The chapter concludes with a justification of the technology stack, linking each technology choice to the requirements established earlier. Together, these artefacts constitute a comprehensive blueprint for the implementation work described in Chapter Four.

### 3.2 System Requirements Specification

#### 3.2.1 Functional Requirements

The functional requirements of the Smart Attendance System describe what the system must do in order to fulfil its purpose. These requirements were derived from an analysis of GCTU's operational context, the institutional policies described in Chapter One, and the gaps identified in Chapter Two. They are organised according to the major subsystems and user roles within the platform.

**User Registration and Authentication.** The system must support two distinct user roles: Student and Lecturer. A user must be able to create an account by providing their full name, institutional email address, a unique identifier appropriate to their role (a student identification number for students, or a staff identification number for lecturers), and a password. Upon registration, the system must securely store the user's credentials and issue a session token that allows subsequent authenticated access. The system must also provide a login interface through which registered users can authenticate using their email address and password, after which the system issues a JSON Web Token valid for twenty-four hours. Authentication must be enforced on all protected resources, and the system must be capable of distinguishing between the student and lecturer roles, granting each role access only to the functions appropriate to that role.

**Course Management.** The system must allow a lecturer to create courses by specifying a course code, course name, venue, scheduled day of the week, and start and end times. Upon creation, the system must automatically generate a unique, human-readable join code for each course, constructed from the course code and a random sequence of characters selected from an unambiguous character set (excluding characters such as zero, the letter O, one, the letter I, and the letter L, which are easily confused). This join code serves as the mechanism through which students enrol in a course. The system must allow a lecturer to view all courses they have created, update course details, remove students from a course, and delete a course. Students must be able to preview a course by entering its join code before committing to enrolment, and must be able to formally enrol by confirming the join action. The system must prevent a student from enrolling in the same course more than once.

**Session Management.** The system must allow a lecturer to initiate an attendance session for a specific course. When creating a session, the lecturer's device must capture the device's current GPS coordinates to anchor the geofence for that session. The system must validate that the captured GPS reading is sufficiently accurate — specifically, that the reported horizontal accuracy is within two hundred metres — and that the location reading is not stale, rejecting readings captured more than two minutes before the session is submitted. The lecturer must be able to specify the duration of the session in minutes and optionally adjust the geofence radius. The system must enforce that only one active session can exist for a given course at any time. Sessions must have a defined lifecycle: they are created in an active state, transition to an ended state either when the lecturer manually ends them or when a scheduled background process detects that the session's duration has elapsed. Upon session end, the system must automatically identify all enrolled students who did not record attendance during the session and create corresponding absence records for them.

**Dynamic QR Code Generation.** The system must generate a unique, time-sensitive QR code token at the start of each attendance session. The token must be encoded in a format that carries all information required for validation, including the session token, the course identifier, the course code, the lecturer's GPS coordinates, the lecturer's GPS accuracy reading, and the configured geofence radius. The system must support automatic token rotation: at regular intervals of thirty seconds, the system must generate a new token and retire the previous one, storing it temporarily as the previous token. This rotation mechanism prevents students from capturing and sharing a static QR code, since any shared code becomes invalid within thirty seconds. To accommodate legitimate students who scan a code in the final seconds of a rotation cycle, the system must accept both the current token and the immediately preceding token during validation. Additionally, the system must accept a token whose embedded timestamp falls within ninety seconds of the validation time, providing a further safety margin for network latency.

**GPS-Based Geofence Verification.** The system must verify that a student's device is physically located within the designated geofence boundary at the time of attendance submission. The boundary is centred on the GPS coordinates captured from the lecturer's device at session creation and has a default radius of fifty metres. To account for the inherent inaccuracy of consumer GPS hardware — which varies significantly between devices and environmental conditions — the system must apply a dynamic buffer to the geofence radius. This buffer is computed as the minimum of one hundred metres and the maximum of twenty metres or the sum of the lecturer's reported GPS accuracy and the student's reported GPS accuracy. The effective geofence radius for any given attendance attempt is therefore the configured base radius plus this dynamic buffer. The geofence calculation must be performed using the Haversine formula, which computes great-circle distances on a spherical Earth model with a radius of 6,371,000 metres. The validation must be performed both on the student's device, to provide immediate feedback before the request is transmitted, and independently on the server, to prevent circumvention through client-side manipulation.

**Attendance Recording.** The system must record a confirmed attendance event when a student successfully passes both QR token validation and GPS geofence verification. The attendance record must capture the session and student identifiers, the verification method used, the computed distance between the student and the lecturer at the time of verification, the student's GPS coordinates and accuracy reading, and the precise timestamp at which attendance was marked. The system must enforce that a student cannot record attendance for the same session more than once. Attendance records must be immediately persisted in the database and must trigger a real-time notification to the lecturer's active session interface.

**Real-Time Monitoring.** The system must provide the lecturer with a live view of attendance as it is recorded during an active session. When a student successfully marks attendance, the lecturer's interface must update in real time, displaying the student's name, identifier, and the time at which they checked in, without requiring the lecturer to manually refresh the page. The system must also notify students on their mobile devices when a live session for one of their enrolled courses begins, allowing them to open the scanner promptly.

**Attendance History and Reporting.** Students must be able to view a personal attendance history showing all sessions they have attended or been marked absent from, with filtering by attendance status. Lecturers must be able to access a complete history of past sessions across all their courses, view a per-session summary including the total number of students, the number present, the number absent, and the overall attendance rate. The system must also provide an analytics view that aggregates attendance data across courses and time periods, identifies students whose attendance falls below seventy-five percent, and presents attendance trends through visual charts. Lecturers must be able to export session attendance data as a comma-separated values file for use in external record systems.

**Health and System Status.** The system must expose a health-check endpoint that verifies database connectivity and reports the server's operational uptime. This endpoint supports monitoring and operational visibility for administrators.

#### 3.2.2 Non-Functional Requirements

The non-functional requirements define the quality attributes that the system must exhibit in order to be fit for deployment in a university environment serving thousands of concurrent users.

**Security.** All user passwords must be stored as one-way cryptographic hashes using the bcrypt algorithm with a work factor of twelve salt rounds, ensuring that raw passwords are never persisted and that brute-force attacks on the stored hashes are computationally expensive. All API communication must be authenticated using JSON Web Tokens signed with a secret key known only to the server. Role-based access control must be enforced on every protected endpoint, preventing students from accessing lecturer functions and vice versa. The QR token rotation mechanism must ensure that any intercepted or photographed QR code becomes invalid within thirty seconds, directly mitigating proxy attendance fraud. The dual-validation of GPS coordinates — on both the client and the server — ensures that geofence verification cannot be bypassed by modifying client-side application behaviour.

**Performance.** The attendance marking endpoint, which is the most latency-sensitive operation in the system, must return a response quickly enough for the student to receive confirmation within a normal scanning interaction. The server-side Haversine calculation is a constant-time mathematical operation that imposes negligible computational overhead. The use of a connection-pooled PostgreSQL driver ensures that database queries are executed efficiently under concurrent load. The student mobile application must be capable of scanning and parsing a QR code in real time using the device camera without perceptible lag.

**Scalability.** The system is designed for a student population of approximately 16,000 across eight campuses, with the potential for many concurrent attendance sessions during peak lecture periods. The NestJS backend is built on a non-blocking, event-driven architecture that handles concurrent requests efficiently. The PostgreSQL relational database supports concurrent reads and writes through its transactional model, and the unique constraints on the attendance and absence tables prevent race conditions when multiple students attempt to record attendance simultaneously. The WebSocket gateway uses a room-based architecture in which students and lecturers subscribe to course-specific channels, ensuring that real-time events are broadcast only to relevant recipients rather than to all connected clients.

**Usability.** The student mobile application must be operable by a student with no prior training beyond standard smartphone usage. The attendance flow — from receiving a session notification to receiving a confirmation of attendance — must involve no more than three user interactions: opening the scanner, pointing the camera at the QR code, and waiting for GPS verification to complete. The lecturer web dashboard must allow a lecturer to create a session and display the live QR code within a single guided workflow. Both interfaces must support dark and light visual themes and must adhere to accessibility contrast ratios consistent with WCAG AA guidelines. The mobile application's design system specifies a minimum touch target size of forty-eight pixels to accommodate users on various screen sizes.

**Availability.** The system must be operational throughout the university's scheduled academic hours. The backend's scheduled session expiry task runs on a one-minute cron cycle, ensuring that sessions that overrun their duration are automatically closed without manual intervention, even if the lecturer's browser session is closed. The WebSocket connections include reconnection logic so that brief network interruptions do not permanently disconnect the real-time monitoring interface.

**Reliability.** The system must prevent duplicate attendance records through database-level unique constraints, ensuring that even if a student's device submits the same attendance request more than once due to a network retry, only one record is persisted. Absence records must be reliably created for all non-attending students when a session ends, regardless of whether the session was ended manually by the lecturer or automatically by the scheduler. The system must handle and communicate meaningful error states to the user interface, distinguishing between common failure scenarios such as an expired QR code, a student located outside the geofence boundary, a duplicate attendance attempt, and a network connectivity failure.

### 3.3 System Architecture

The Smart Attendance System follows a three-tier client-server architecture, separating concerns across a presentation tier, an application logic tier, and a data tier. The system comprises three distinct client applications — the student mobile application and the lecturer web dashboard as presentation-tier clients — and a single shared backend server at the application logic tier, which in turn communicates with a PostgreSQL relational database at the data tier.

The backend server is implemented using the NestJS framework running on the Node.js runtime, and is organised into discrete feature modules, each encapsulating the routes, business logic, and data access logic for a specific domain. The Auth module handles user registration, credential validation, JWT issuance, and authentication middleware. The Courses module manages the full lifecycle of course entities, including creation, modification, student enrolment, and deletion. The Sessions module governs the creation and expiry of attendance sessions, QR token generation and rotation, and the scheduling of automatic session closure through a cron-based task that executes once per minute. The Attendance module is responsible for processing attendance submission requests, performing QR token validation and GPS geofence verification, persisting attendance and absence records, and exposing attendance history to authorised clients. The Events module hosts the WebSocket gateway, which maintains persistent bidirectional connections with all active client applications and broadcasts named events to course-specific rooms as attendance-related state changes occur.

The backend exposes all its functionality through a RESTful HTTP API, with all routes prefixed under the `/api` namespace. The API is consumed by both the student mobile application and the lecturer web dashboard through typed API client libraries maintained within each frontend codebase. In addition to the REST API, the backend maintains WebSocket connections with connected clients using the Socket.IO library, enabling server-to-client push notifications for real-time events such as session start, session end, QR token refresh, and new attendance check-in.

The student mobile application is a React Native application built with the Expo framework, targeting both Android and iOS platforms. It communicates with the backend exclusively over HTTP and WebSocket connections. The application uses file-based routing provided by the expo-router library, and wraps all screens in a hierarchy of context providers that manage authentication state, attendance records, and live session subscriptions. Authentication tokens are persisted securely using the expo-secure-store library, which stores data in the platform's hardware-backed secure enclave on supported devices.

The lecturer web dashboard is a React single-page application built with the Vite build tool, served as static assets in a browser environment. It communicates with the same backend API and WebSocket server as the mobile application. Authentication tokens are stored in the browser's localStorage. The dashboard is structured around a fixed sidebar navigation and a central content area rendered through React Router's outlet mechanism, allowing navigation between pages without a full page reload.

The data tier consists of a single PostgreSQL database accessed through the Prisma ORM. Prisma provides a type-safe query builder and manages the database schema through a declarative schema definition file, from which it generates both the SQL migration scripts and the TypeScript client bindings used in the backend services. The database stores all persistent entities in the system: users, courses, enrolments, sessions, attendance records, and absence records.

The planned blockchain layer, to be integrated in a subsequent development phase, will sit alongside the existing database as an additional persistence mechanism. Its role will be to receive a cryptographic summary of each finalised attendance session and record it immutably on a distributed ledger, providing an externally verifiable audit trail that cannot be altered even by a database administrator. The current system includes user interface scaffolding for this feature — a dedicated blockchain ledger page and a settings toggle for enabling blockchain writes — but the backend integration with an actual blockchain network has not yet been implemented.

The overall data flow through the system during an attendance event proceeds as follows. The lecturer's browser application captures GPS coordinates and issues a session creation request to the backend REST API. The backend validates the request, generates a QR token, persists the session in the database, and broadcasts a session-started event over WebSocket to all students enrolled in the course. Students receive a push notification on their mobile devices and open the QR scanner. The scanner decodes the QR payload from the lecturer's displayed QR code and passes the extracted parameters to the GPS verification screen. The student's device captures its current GPS coordinates, performs a client-side Haversine distance calculation, and if the student is within the effective geofence, transmits an attendance submission request to the backend REST API. The backend independently repeats the token validation and Haversine geofence calculation, and if both pass, persists an attendance record and broadcasts an attendance-new event over WebSocket. The lecturer's active session interface receives this event and updates the live check-in list in real time.

An architecture diagram for this system would depict the following arrangement. At the centre is the NestJS backend server, connected on one side to the PostgreSQL database (via the Prisma ORM) and on the other side to the planned blockchain node (labelled as a future component). Above the backend, two clients connect via HTTP REST API arrows and bidirectional WebSocket arrows: the React Native student mobile application on the left and the React lecturer web dashboard on the right. Within the backend, the five modules (Auth, Courses, Sessions, Attendance, Events/WebSocket Gateway) would be shown as internal boxes within the backend boundary. The student mobile application would internally show the three key screens involved in an attendance flow: the QR Scanner, the GPS Verify screen, and the Attendance Confirmed screen. The lecturer web dashboard would show the Create Session, Active Session, and Live Monitor screens. Arrows from the lecturer dashboard to the backend would carry labels such as "POST /api/sessions" and "PATCH /api/sessions/:id/refresh-qr", while arrows from the student app would carry labels such as "POST /api/attendance/mark". WebSocket event labels ("session:started", "attendance:new") would appear on the bidirectional WebSocket arrows.

### 3.4 Use Case Design

The system identifies three primary actors whose interactions drive all system behaviour: the Student, the Lecturer, and the System (representing automated processes triggered by time or events rather than by direct user action). The roles of each actor and the use cases belonging to each are described below.

The **Student** actor represents an enrolled GCTU student using the mobile application. A student interacts with the system to manage their own account, enrol in courses, and record their attendance. The use cases associated with the student actor are as follows. The student can register a new account by providing their name, email, student identification number, and password. The student can log in using their credentials and log out to end their session. The student can view a list of their enrolled courses on the home dashboard. The student can join a new course by entering the course's join code; before confirming enrolment, the student can preview the course details. The student can scan a QR code using the device camera during a live session, which triggers the attendance registration flow. The student can view their GPS location verification status while the application confirms their physical proximity to the lecture venue. The student can view a confirmation screen upon successful attendance registration. The student can view their complete attendance history, filtered by present or absent status. The student can manage their profile settings, including toggling location service preferences and notification preferences.

The **Lecturer** actor represents a GCTU lecturer using the web dashboard. The lecturer is responsible for managing the academic structure that the attendance system operates within. The use cases associated with the lecturer actor are as follows. The lecturer can register an account and log in and out of the web dashboard. The lecturer can create a new course, specifying its code, name, venue, and schedule. The lecturer can view all courses they manage and edit or delete existing courses. The lecturer can view the list of students enrolled in each course and remove a student from a course if necessary. The lecturer can initiate an attendance session for a specific course by capturing their current GPS location, selecting session parameters, and submitting the session creation form. The lecturer can view a live QR code displayed on the active session screen and present it to students for scanning. The lecturer can monitor real-time check-in activity during a live session, observing each student's check-in as it occurs. The lecturer can manually end a session before its scheduled expiry. The lecturer can view a summary of a completed session, showing who attended and who was absent. The lecturer can browse the history of all past sessions. The lecturer can view analytical reports on attendance trends across their courses. The lecturer can download a session's attendance record as a CSV file. The lecturer can view the blockchain ledger page, which in the current phase displays a placeholder for the planned immutable record log.

The **System** actor represents automated behaviour that occurs independently of direct user action. The system automatically rotates the QR token every thirty seconds during an active session, replacing the current token with a newly generated one and retaining the previous token for a brief grace period. The system automatically expires sessions whose duration has elapsed, transitioning them from active to ended status and creating absence records for all enrolled students who did not register attendance. The system broadcasts WebSocket events to notify connected clients when a session starts, when a session ends, when the QR code is refreshed, and when a new attendance record is created.

A use case diagram for this system would be structured as follows. Three actor stick figures labelled "Student", "Lecturer", and "System" are positioned outside the system boundary rectangle. Inside the boundary, oval use case nodes are drawn and connected to their respective actors with association lines. The student actor connects to: Register Account, Login, Logout, View Enrolled Courses, Join Course (which includes Preview Course as a subordinate use case in an include relationship), Scan QR Code (which includes GPS Verification as an include), View Attendance Confirmation, View Attendance History, and Manage Profile Settings. The lecturer actor connects to: Register Account, Login, Logout, Create Course, View/Edit/Delete Course, Manage Enrolled Students, Create Attendance Session (which includes Capture GPS Location as an include), View Live QR Code, Monitor Real-Time Check-ins, End Session Manually, View Session Summary, View Session History, View Analytics Reports, Download CSV Report, and View Blockchain Ledger. The system actor connects to: Auto-Rotate QR Token, Auto-Expire Session (which includes Create Absence Records as an include), and Broadcast WebSocket Events. A dashed extend arrow runs from the GPS Verification use case back to Scan QR Code, and a dashed include arrow connects Create Attendance Session to Capture GPS Location.

### 3.5 System Design

#### 3.5.1 Context Diagram (DFD Level 0)

The context diagram represents the Smart Attendance System as a single process and illustrates all external entities that interact with it, along with the data flows that cross the system boundary. At this level of abstraction, the internal workings of the system are not shown; the diagram presents only what enters and leaves the system.

The context diagram contains the following elements. The central process is a single circle labelled "Smart Attendance System". Four external entities surround this central process. The first external entity is the **Student**, represented as a rectangle to the left of the central process. The second external entity is the **Lecturer**, represented as a rectangle to the right. The third external entity is the **GPS Satellite Network**, which represents the external location infrastructure that both the student's mobile device and the lecturer's browser consult when capturing coordinates. The fourth external entity is the **Blockchain Network**, representing the planned external distributed ledger (shown with a dashed border or a future-phase annotation to indicate that this integration is not yet implemented).

The data flows are as follows. From the Student to the system: Registration Details (name, email, student ID, password), Login Credentials (email, password), Course Join Code, QR Code Scan Data (the decoded JSON payload from the QR image), GPS Location Data (latitude, longitude, accuracy, captured from the device), and Attendance Submission Request. From the system to the Student: Authentication Token, Session Started Notification (push notification), Enrolled Course List, Attendance Confirmation, Attendance History Records, and Error Responses (for cases such as outside geofence, expired token, duplicate attendance).

From the Lecturer to the system: Registration Details, Login Credentials, Course Details (code, name, venue, schedule), Session Parameters (course, duration, geofence radius), Lecturer GPS Location Data, QR Refresh Request, and Session End Request. From the system to the Lecturer: Authentication Token, Generated Join Code, Live QR Code Data, Real-Time Check-in Events, Session Summary Data, Attendance History Data, Analytics Report Data, and CSV Export File.

From the GPS Satellite Network to the system (via device firmware): GPS Coordinates and Accuracy Readings (these are captured by the device operating system and passed to the application; the system does not communicate directly with satellites but depends on this external infrastructure for valid location data).

From the system to the Blockchain Network: Finalised Attendance Record Hash (a cryptographic summary of a closed session's attendance data, to be written to the ledger — this flow is planned for the next phase). From the Blockchain Network to the system: Transaction Confirmation and Block Reference (returned after a successful ledger write — also planned).

#### 3.5.2 DFD Diagram 0 (Level 1)

At Level 1, the Smart Attendance System is decomposed into its major internal processes, and the data stores that these processes read from and write to are introduced. The external entities remain the same as in the context diagram, and all data flows crossing the system boundary are preserved; the diagram adds detail about which internal process handles each flow.

The Level 1 diagram contains the following processes, data stores, and data flows.

**Processes:** Process 1 — User Authentication and Account Management; Process 2 — Course and Enrolment Management; Process 3 — Session and QR Code Management; Process 4 — Attendance Verification and Recording; Process 5 — Reporting and History; Process 6 — Real-Time Event Distribution (WebSocket Gateway); Process 7 — Automated Session Lifecycle Management (Scheduler).

**Data Stores:** D1 — Users; D2 — Courses; D3 — Enrolments; D4 — Sessions; D5 — Attendance Records; D6 — Absence Records.

**Data Flows:** The Student sends Registration Details and Login Credentials to Process 1. Process 1 reads from and writes to D1 (Users). Process 1 returns an Authentication Token to the Student and the Lecturer. The Lecturer sends Course Details to Process 2. Process 2 reads and writes D2 (Courses) and D3 (Enrolments). Process 2 returns a Join Code to the Lecturer. The Student sends a Course Join Code to Process 2, which validates it against D2 and creates an entry in D3. The Lecturer sends GPS Location Data and Session Parameters to Process 3. Process 3 validates the data, generates a QR token, and writes a new record to D4 (Sessions). Process 3 sends QR Code Data to the Lecturer interface and sends a Session Started notification to Process 6. Process 3 also receives QR Refresh Requests from the Lecturer and writes updated tokens back to D4. The Student sends QR Scan Data and GPS Location Data to Process 4. Process 4 reads D4 (Sessions) to retrieve the current and previous QR tokens, validates the scanned token, reads D3 (Enrolments) to confirm course membership, reads D5 (Attendance Records) to check for duplicates, performs the Haversine geofence calculation, and writes a new record to D5 on success. Process 4 sends an Attendance Confirmation to the Student and sends an Attendance New event to Process 6. The Lecturer and Student send data requests to Process 5. Process 5 reads from D4, D5, and D6 to generate history views, session summaries, and analytics reports, which it returns to the requesting actor. Process 5 also prepares CSV export files for the Lecturer. Process 6 receives events from Processes 3 and 4 and distributes them to the appropriate connected clients (lecturers and students) based on course-room subscriptions. Process 7 runs on a one-minute cycle, reads D4 (Sessions) for active sessions past their duration, transitions their status to ended, reads D3 (Enrolments) and D5 (Attendance Records) to identify non-attending students, and writes records to D6 (Absence Records).

#### 3.5.3 DFD Level 2 Diagrams

**Level 2: QR Code Generation and Validation (Expansion of Process 3)**

Process 3 is decomposed into the following sub-processes. Sub-process 3.1 — GPS Validation: receives the lecturer's submitted GPS coordinates and accuracy reading, checks that the accuracy value does not exceed two hundred metres, and checks that the timestamp of the location reading is within two minutes of the current server time. If validation fails, a rejection response is returned to the Lecturer. If validation passes, the validated location data is passed to Sub-process 3.2. Sub-process 3.2 — Session Existence Check: reads D4 (Sessions) to confirm that no active session already exists for the selected course. If an active session exists, a rejection is returned; otherwise, execution proceeds to Sub-process 3.3. Sub-process 3.3 — QR Token Generation: generates a token string in the format `SA-{unix_timestamp_seconds}-{6_random_hex_chars}`, encodes it into a JSON payload alongside the course identifier, course code, lecturer GPS coordinates, lecturer GPS accuracy, and configured geofence radius, and passes the complete session data to Sub-process 3.4. Sub-process 3.4 — Session Persistence: writes the new session record to D4, including the token, previous token (null at creation), GPS coordinates, accuracy, geofence radius, duration, start time, and active status. Sub-process 3.5 — QR Rotation: is triggered either by a lecturer-initiated refresh request from the web dashboard or, at the client level, by a thirty-second timer. It reads the current token from D4, stores it as the previous token, generates a new token using the same method as Sub-process 3.3, and writes the updated token pair back to D4. It then emits a QR Refreshed event to Sub-process 3.6. Sub-process 3.6 — Event Emission: receives session lifecycle events (session created, QR refreshed, session ended) and passes them to Process 6 (WebSocket Gateway) for distribution.

**Level 2: GPS Geofence Verification (Expansion of Process 4, GPS sub-flow)**

The GPS verification component of Process 4 is decomposed as follows. Sub-process 4.1 — Student Location Capture: the student's device requests location permission and calls the platform location API with Balanced accuracy mode, receiving a latitude, longitude, and accuracy value. Sub-process 4.2 — QR Payload Extraction: the scanned QR code is decoded from JSON, extracting the venue's latitude and longitude (from the lecturer's coordinates stored in the QR payload), the lecturer's accuracy value, and the configured geofence radius. Sub-process 4.3 — Dynamic Buffer Calculation: computes the dynamic GPS buffer as `min(max(lecturerAccuracy + studentAccuracy, 20), 100)` metres, producing an effective geofence radius by adding this buffer to the base geofence radius. Sub-process 4.4 — Haversine Distance Calculation: applies the Haversine formula using the student's captured latitude and longitude as one point and the venue's latitude and longitude as the other, yielding a great-circle distance in metres. Sub-process 4.5 — Proximity Decision: compares the computed distance to the effective geofence radius. If the distance is within the radius, execution proceeds to attendance submission; if not, a "outside geofence" rejection is returned to the student. Sub-process 4.6 — Server-Side Re-Validation: the backend independently repeats Sub-processes 4.3 and 4.4 using the coordinates submitted in the attendance request and the venue coordinates stored in D4, ensuring that the client's reported GPS data has not been manipulated.

**Level 2: Attendance Recording (Expansion of Process 4, recording sub-flow)**

Sub-process 4.7 — Token Format Check: verifies that the submitted token begins with the `SA-` prefix, confirming that it originated from this system. Sub-process 4.8 — Active Session Lookup: reads D4 (Sessions) using the submitted course identifier, retrieving the active session record including its current token, previous token, geofence parameters, and status. If no active session is found, a rejection is returned. Sub-process 4.9 — Expiry Check: confirms that the session's start time plus its duration has not passed, ensuring the session is still within its scheduled window. Sub-process 4.10 — Token Validity Check: applies a three-condition acceptance rule. The submitted token is accepted if it exactly matches the session's current QR token; or if it matches the session's stored previous token; or if the Unix timestamp embedded in the token (extracted as the second segment of the `SA-{timestamp}-{hex}` format) falls within ninety seconds of the current server time. If none of these conditions is satisfied, a "token expired or invalid" rejection is returned. Sub-process 4.11 — Enrolment Check: reads D3 (Enrolments) to verify that the submitting student is enrolled in the course to which the session belongs. Sub-process 4.12 — Duplicate Check: reads D5 (Attendance Records) for any existing record with the same session and student identifiers. If a duplicate is found, a "already recorded" rejection is returned. Sub-process 4.13 — Attendance Persistence: writes a new record to D5 (Attendance Records) containing the session ID, student ID, verification method (QR_GPS), computed distance, student coordinates, accuracy, and the current timestamp. Sub-process 4.14 — Event Emission: passes an attendance-new event containing the student's details to Process 6 for real-time broadcast to the lecturer's interface.

#### 3.5.4 Flowcharts

**Flowchart 1: Student Attendance Registration**

The student attendance registration process begins when the student receives a push notification on their mobile device indicating that a lecture session has started. The first decision point asks whether the student has granted the application permission to use the device camera. If not, the application prompts the user to enable camera access in the device settings, and the flow waits. Once camera permission is confirmed, the QR scanner screen opens and the device camera activates. The student points the camera at the QR code displayed on the lecturer's screen. The application's barcode scanner library detects and decodes the QR code from the camera frame. The next step verifies that the decoded string begins with the `SA-` prefix to confirm it is a valid system-generated code. If the prefix is not found, an error message is shown and the scanner remains active. If the prefix is confirmed, the application parses the JSON payload to extract the token, course identifier, lecturer coordinates, lecturer accuracy, and geofence radius. The application then calls the backend's check-attendance endpoint to determine whether the student has already recorded attendance for this session. If attendance has already been recorded, a "already marked" message is displayed and the flow ends. If not, the flow proceeds to GPS verification. The application checks whether location permission has been granted. If not, a permission request dialog is shown; if the user denies, an "access denied" error is shown and the flow ends. If permission is granted, the device captures the current GPS position, obtaining a latitude, longitude, and accuracy value. The application computes the dynamic GPS buffer, adds it to the base geofence radius from the QR payload, and applies the Haversine formula to calculate the distance between the student's location and the lecturer's location. The next decision asks whether this distance is less than or equal to the effective geofence radius. If not, an animated "outside the venue" failure screen is shown and the flow ends. If the student is within the geofence, the application submits an attendance marking request to the backend, including the QR token, course identifier, and GPS data. The backend performs its own validation sequence (token, enrolment, duplicate, geofence). If any server-side check fails, the corresponding error message is returned and displayed to the student. If all checks pass, the backend creates the attendance record and returns a success response. The application displays an animated attendance confirmation screen showing the course name, date, time, and a "QR + GPS Verified" badge. The flow ends.

**Flowchart 2: Lecturer Session Creation and QR Code Generation**

The lecturer session creation flow begins when the lecturer navigates to the "Create Session" page on the web dashboard. The page initiates automatic GPS capture from the browser's Geolocation API, collecting three location samples and averaging them to produce a single representative coordinate set. The first decision asks whether the GPS capture succeeded within thirty seconds. If it timed out or was denied, the lecturer is informed and prompted to retry or to check their browser location permissions. If GPS capture succeeded, the accuracy value of the captured position is evaluated. If the accuracy exceeds two hundred metres, a warning is displayed advising the lecturer to wait for a more accurate reading or to move to an area with better GPS reception. Assuming an acceptable accuracy, the system check panel displays confirmation that the backend is reachable and GPS coordinates are ready. The lecturer selects the course from a dropdown of their active courses, sets the session duration using the duration picker, and optionally adjusts the geofence radius. The lecturer confirms the session by clicking "Start Session". The application submits a POST request to the backend sessions endpoint with the course identifier, duration, geofence radius, GPS coordinates, and accuracy. The backend validates the GPS accuracy and staleness of the coordinates. If the coordinates are stale (captured more than two minutes ago), a rejection is returned and the lecturer is prompted to recapture. If valid, the backend checks whether an active session already exists for the chosen course. If one exists, a rejection is returned. If not, the backend generates a QR token and persists the session. The backend broadcasts a session-started WebSocket event to all students in the course room. The lecturer dashboard navigates to the Active Session page, which renders the QR code using the QR code library. A thirty-second countdown timer starts. When the timer reaches zero, the application calls the QR refresh endpoint, the backend generates a new token (storing the previous one), and the QR code on screen updates. This thirty-second cycle repeats until the session ends. The lecturer can click "End Session" at any time, which triggers a confirmation dialog. Upon confirmation, a PATCH request is sent to end the session. The backend transitions the session to ended status, generates absence records for non-attending students, and broadcasts a session-ended WebSocket event. The dashboard navigates to the Session Summary page. The flow ends.

**Flowchart 3: User Login and Authentication**

The login flow begins when the user opens the login screen on either the student mobile application or the lecturer web dashboard. The user enters their email address and password and submits the form. The application performs basic client-side validation: it checks that the email field is not empty and that the password field is not empty. If either field is empty, an inline validation message is displayed and the form is not submitted. If both fields contain values, the application submits a POST request to the backend's login endpoint containing the email and password. The backend locates the user record by email address in the Users data store. If no record is found, a "invalid credentials" error response is returned. If a user record is found, the backend compares the submitted password against the stored password hash using bcrypt's comparison function. If the comparison fails, the same "invalid credentials" error is returned (the identical error message for both cases prevents an attacker from determining whether an email address is registered). If the password matches, the backend generates a JWT signed with the server's secret, set to expire in twenty-four hours, containing the user's identifier and role. The JWT is returned to the application in the response body. The application stores the token securely: the student mobile application writes it to the platform's secure store using expo-secure-store; the web dashboard writes it to the browser's localStorage. The application then calls the "get me" endpoint to retrieve the user's full profile. The application stores the user profile in the authentication context and navigates the user to their role-appropriate home screen: students are directed to the home tab with their enrolled courses, and lecturers are directed to the dashboard overview page. The flow ends.

**Flowchart 4: Report Generation**

The report generation flow begins when the lecturer navigates to the Reports page on the web dashboard. The page immediately fetches all session records associated with the lecturer's courses from the backend. The data is processed to compute aggregate statistics: the average attendance rate across all sessions, the total number of sessions conducted, and the percentage of attendance records that included GPS verification. These figures are displayed as key performance indicator cards at the top of the page. The application constructs an attendance trend dataset by grouping sessions by date and computing the average attendance rate for each date. This dataset is passed to the recharts AreaChart component to render a visual trend line. The lecturer can apply filters: selecting a specific course from a dropdown, or selecting a time period (last seven days, last thirty days, or the full semester). When a filter is applied, the application re-queries the fetched data locally, recomputes the aggregate statistics, and re-renders the charts and tables without a new network request. The course performance table is populated by iterating over the lecturer's courses and computing per-course statistics: average attendance rate, total sessions, trend direction (improving or declining relative to the previous period), and a miniature sparkline chart. The flagged students list is computed by identifying enrolled students whose attendance rate across all sessions falls below seventy-five percent; their names, student identifiers, course codes, and attendance percentages are listed. If the lecturer wishes to export a specific session's data, they navigate to the History page, locate the session row, and click the download icon. The application calls the session summary endpoint, receives the full attendance list, serialises it as comma-separated values, and triggers a browser file download. The flow ends.

#### 3.5.5 Entity Relationship Diagram (ERD)

The database schema comprises six entities whose relationships govern all persistent data in the system. The entities, their attributes, and the relationships between them are described below in sufficient detail for the diagram to be drawn accurately.

The **User** entity is the central identity record for all system participants. Its attributes are: `id` (UUID, primary key, auto-generated), `email` (string, unique, not null), `fullName` (string, not null), `passwordHash` (string, not null), `role` (enumeration: STUDENT or LECTURER, not null), `studentId` (string, unique, nullable — populated only for students), `staffId` (string, nullable — populated only for lecturers), `createdAt` (timestamp), and `updatedAt` (timestamp).

The **Course** entity represents an academic course managed by a lecturer. Its attributes are: `id` (UUID, primary key), `courseCode` (string, not null), `courseName` (string, not null), `joinCode` (string, unique, not null), `venue` (string, nullable), `dayOfWeek` (string, nullable), `startTime` (string, nullable), `endTime` (string, nullable), `lecturerId` (UUID, foreign key referencing User.id, not null), `createdAt` (timestamp), and `updatedAt` (timestamp).

The **Enrollment** entity is a junction table implementing the many-to-many relationship between students and courses. Its attributes are: `id` (UUID, primary key), `studentId` (UUID, foreign key referencing User.id, not null), `courseId` (UUID, foreign key referencing Course.id, not null), and `enrolledAt` (timestamp). A composite unique constraint exists on the combination of `studentId` and `courseId`, preventing duplicate enrolments.

The **Session** entity represents a single attendance session created by a lecturer. Its attributes are: `id` (UUID, primary key), `courseId` (UUID, foreign key referencing Course.id, not null), `qrToken` (string, not null), `previousQrToken` (string, nullable), `lat` (floating-point number, nullable), `lng` (floating-point number, nullable), `lecturerAccuracy` (floating-point number, nullable), `geofenceRadius` (integer, not null, default 50), `duration` (integer, in minutes, not null), `startedAt` (timestamp, not null), `endedAt` (timestamp, nullable), `status` (enumeration: ACTIVE or ENDED, not null), and `createdAt` (timestamp).

The **Attendance** entity records a confirmed attendance event for a student in a session. Its attributes are: `id` (UUID, primary key), `sessionId` (UUID, foreign key referencing Session.id, not null), `studentId` (UUID, foreign key referencing User.id, not null), `method` (enumeration: QR_GPS, not null), `distance` (floating-point number, nullable — the computed Haversine distance in metres), `latitude` (floating-point number, nullable), `longitude` (floating-point number, nullable), `accuracy` (floating-point number, nullable), and `markedAt` (timestamp, not null). A composite unique constraint exists on the combination of `sessionId` and `studentId`.

The **Absence** entity records an automatically generated non-attendance event for a student in a session. Its attributes are: `id` (UUID, primary key), `sessionId` (UUID, foreign key referencing Session.id, not null), `studentId` (UUID, foreign key referencing User.id, not null), and `createdAt` (timestamp). A composite unique constraint exists on the combination of `sessionId` and `studentId`.

The relationships between entities are as follows. Between **User** and **Course**: a one-to-many relationship in which one User (in the Lecturer role) may create many Courses, but each Course is owned by exactly one Lecturer. This is implemented via the `lecturerId` foreign key on Course. Between **User** and **Enrollment**: a one-to-many relationship in which one User (in the Student role) may have many Enrollment records (one per enrolled course). Between **Course** and **Enrollment**: a one-to-many relationship in which one Course may have many Enrollment records. Together, the User-Enrollment-Course relationships form a many-to-many association between students and courses, mediated by the Enrollment junction table. Between **Course** and **Session**: a one-to-many relationship in which one Course may have many Sessions over its lifetime, but each Session belongs to exactly one Course. Between **Session** and **Attendance**: a one-to-many relationship in which one Session may have many Attendance records (one per present student), and each Attendance record belongs to exactly one Session. Between **User** and **Attendance**: a one-to-many relationship in which one User (Student) may have many Attendance records across sessions. Between **Session** and **Absence**: a one-to-many relationship in which one Session may have many Absence records, and each Absence record belongs to exactly one Session. Between **User** and **Absence**: a one-to-many relationship in which one User (Student) may have many Absence records.

An ERD drawn from this description would show six rectangular entity boxes. The User box is connected to the Course box by a line labelled "creates (1 to many)", to the Enrollment box by "has (1 to many)", and to both the Attendance and Absence boxes by "records (1 to many)" lines. The Course box is connected to the Enrollment box by "has (1 to many)" and to the Session box by "has (1 to many)". The Session box connects to both the Attendance and Absence boxes by "generates (1 to many)" lines. Crow's-foot notation would be used at each endpoint to indicate cardinality.

#### 3.5.6 Interface Design

**Student Mobile Application**

The student mobile application presents a consistent visual identity throughout, drawing from a design system that uses GCTU's institutional colours — a deep blue (`#003366`) and a gold (`#C5960C`) — alongside a full set of semantic colour tokens supporting both light and dark themes. Typography uses a size scale from twelve to twenty-eight point with weight variation to establish visual hierarchy, and all interactive elements meet the minimum touch target size of forty-eight pixels.

The **Splash Screen** is the first screen displayed when the application launches. It presents the GCTU crest at the centre of the screen against a white background, accompanied by the university name and the application title "Smart Attendance". The screen is non-interactive; after a brief pause of approximately one and a half seconds, the application checks for a stored authentication token and navigates automatically to either the home screen (if a valid session exists) or the login screen (if not).

The **Login Screen** presents a full-screen layout with the GCTU crest at the top, followed by the heading "Welcome Back" and a subheading indicating that GPS will be used for attendance verification. Below the headings are two labelled text input fields for email address and password. A "Sign In" primary button submits the credentials. A link at the bottom navigates to the registration screen. The **Registration Screen** is divided into two sequential steps to reduce cognitive load. Step One collects the student's full name, email address, and student identification number. A "Continue" button advances to Step Two, which collects the student's password and a confirmation of the password. The password field includes a real-time strength indicator. A "Create Account" primary button submits the full registration payload. A back navigation option on Step Two returns the student to Step One without losing their previously entered data.

The **Home Screen** (the default tab in the tab navigator) presents a greeting with the student's first name, a summary bar showing the total number of enrolled courses, and a prominent "Scan QR Code" action button. Below the action button is a horizontally scrolling section labelled "Live Now" that appears only when at least one enrolled course has an active session; each live session card displays the course code, course name, and a pulsing "Scan Now" badge. Below this section is a vertically stacked list of all enrolled courses, each showing the course code, course name, venue, and scheduled day and time. A "Join Course" text link at the bottom navigates to the join course screen. A pull-to-refresh gesture reloads all data from the server.

The **History Screen** displays the student's complete attendance record as a vertically scrollable list. A row of filter chips at the top allows the student to filter by "All", "Present", or "Absent". Each list item shows the course code, course name, date, time, and a coloured status badge (green for present, red for absent). Items marked as present additionally show the verification method.

A large "Open Scanner" button at the bottom launches the QR scanner screen. This design ensures that the camera is not activated unnecessarily while the student is browsing other tabs.

The **QR Scanner Screen** occupies the full screen with the camera feed. A dark semi-transparent overlay surrounds a square viewfinder area in the centre, indicated by four corner-bracket markers. The **Attendance Confirmed Screen** displays a large animated success icon alongside a "QR + GPS Verified" badge rendered in gold. The course name, course code, date, time, and venue are displayed in a card below the badge. A "Done" button dismisses the screen and returns the student to the home tab.

In the initial state, a text input field accepts the course join code and a "Preview Course" button submits it. In the preview state, the course name, course code, current enrolled student count, and the join code are displayed in a card, with a "Join Course" primary button and a "Go Back" secondary button. In the success state, a confirmation message indicates that the student has successfully joined the course, and a "Go to Courses" button returns to the home screen.

The **Profile Screen** presents the student's avatar (displaying their initials), full name, and email address. Below the identity section, a settings list includes a Location Services toggle (enabling or disabling GPS for attendance), a High Accuracy Mode toggle, and a Notifications toggle. A GPS Status indicator card shows the current state of the device's location services with a colour-coded status label. Device information (platform and application version) is displayed at the bottom. A "Sign Out" destructive button at the bottom of the screen triggers a logout confirmation.

### 3.6 Technology Stack and Justification

The technology choices made in this project were guided by the requirements established in Section 3.2 — specifically the need for a cross-platform mobile student interface, a modern web dashboard for lecturers, a performant and modular backend, reliable real-time event delivery, and a database capable of maintaining strong consistency under concurrent load.

**NestJS (Backend Framework).** The backend is implemented using NestJS version 11, a progressive Node.js framework that enforces a modular, opinionated application structure inspired by Angular. NestJS was selected over alternatives such as Express.js and Fastify for several reasons. First, its module-based architecture enforces a clean separation of concerns, ensuring that the Auth, Courses, Sessions, Attendance, and Events domains are each self-contained and independently maintainable. Second, NestJS provides first-class integration with Passport.js for authentication strategy management, with Socket.IO for WebSocket gateways, and with the `@nestjs/schedule` module for cron-based task scheduling — all of which are requirements of this system. Third, NestJS's use of TypeScript throughout the application layer enables compile-time type checking across the entire backend codebase, reducing a class of runtime errors that would otherwise require extensive integration testing to detect.

**PostgreSQL (Database).** PostgreSQL was selected as the relational database management system for this project. The data model of the attendance system is inherently relational: students are enrolled in courses, sessions belong to courses, attendance records reference both sessions and students, and absence records are derived from the intersection of enrolment and non-attendance. PostgreSQL's support for foreign key constraints, composite unique constraints, and ACID-compliant transactions ensures that the referential integrity of these relationships is enforced at the database level. The composite unique constraints on the Attendance and Absence tables — preventing duplicate records for the same student and session — are a direct implementation of the system's requirement to prevent double-marking, and PostgreSQL enforces these atomically under concurrent load. An alternative such as MongoDB, a document-oriented database, would require the application layer to enforce these constraints, introducing risk. PostgreSQL was therefore the appropriate choice for this domain.

**Prisma (ORM).** Prisma version 7.4.2 was selected as the object-relational mapper bridging the NestJS backend and PostgreSQL. Prisma generates a fully type-safe client from the declarative schema definition file, meaning that every database query written in the backend services is checked against the schema at compile time. This eliminates an entire class of bugs arising from schema-query mismatches. Prisma's migration system generates versioned SQL migration scripts from schema changes, providing a reproducible and auditable history of database schema evolution. The use of the `@prisma/adapter-pg` driver adapter with a native `pg` connection pool ensures that connections are efficiently reused across requests, supporting the concurrent-user performance requirements of the system.

**React Native with Expo (Student Mobile Application).** The student mobile application is built using React Native 0.81.5 with the Expo SDK 54 framework. React Native was chosen because it enables the development of a single codebase that compiles to native applications on both Android and iOS, addressing GCTU's heterogeneous student device population without requiring the development and maintenance of two separate platform-specific codebases. The Expo framework was selected over bare React Native because it provides a managed set of pre-built native modules — including `expo-camera` for QR scanning, `expo-location` for GPS access, `expo-secure-store` for hardware-backed token storage, and `expo-notifications` for local push notifications — each of which corresponds directly to a functional requirement of the student interface. Using Expo's managed workflow eliminates the need to configure native build toolchains for each module individually, significantly reducing development complexity. Expo Router, a file-based routing library built on React Navigation, was used to define the application's navigation structure, resulting in a declarative, filesystem-visible routing scheme that simplifies the addition of new screens.

**React with Vite (Lecturer Web Dashboard).** The lecturer web dashboard is implemented as a React 19 single-page application, built and served using the Vite 6 build tool. React was selected for its component-based model, which maps naturally to the dashboard's compositional structure of reusable cards, charts, tables, and modals. React 19's concurrent rendering capabilities improve perceived responsiveness when the interface is handling simultaneous data fetches and real-time WebSocket updates. Vite was chosen over Create React App as the build tool because it provides near-instant hot module replacement during development through its native ES module server, significantly improving the development iteration cycle. Tailwind CSS 4 was used for styling, enabling consistent, utility-first styles that can be applied directly in JSX without context-switching to separate CSS files.

**Socket.IO (Real-Time Communication).** Socket.IO was selected to implement the bidirectional real-time communication layer between the backend and both client applications. The system's real-time requirements — delivering session-started notifications to students and attendance-new events to lecturers with minimal latency — demand a persistent connection rather than repeated HTTP polling. Socket.IO's room-based broadcasting mechanism, in which clients subscribe to course-specific rooms (identified as `course:{courseId}`), ensures that events are delivered only to clients enrolled in the relevant course, avoiding unnecessary broadcast traffic. Socket.IO's automatic fallback to HTTP long-polling on networks that block WebSocket connections improves reliability in campus environments where network configuration may be restrictive.

**qrcode.react (QR Code Rendering).** The `qrcode.react` library was used to render the session QR code as an SVG element in the lecturer's web dashboard. This library was selected because it renders QR codes natively in the browser without a server round-trip, meaning the QR code updates immediately when the token is refreshed, with no additional latency introduced by a network request for the image. SVG rendering ensures that the QR code remains crisp and scannable at any display size, including the fullscreen projection mode used during lectures.

**expo-camera (QR Code Scanning).** The `expo-camera` library's `CameraView` component with barcode scanning enabled was used in the student mobile application to scan QR codes from the device camera. This library was selected because it is the officially maintained Expo module for camera access, ensuring compatibility with the Expo SDK version and build infrastructure in use. The barcode scanner callback returns the decoded string directly, requiring no additional image processing on the application side.

**expo-location (GPS Access).** The `expo-location` library provides cross-platform access to the device's location services. The `Location.getCurrentPositionAsync` method with `Accuracy.Balanced` was chosen for GPS capture in the student application. Balanced accuracy was preferred over high accuracy because it offers a reasonable trade-off between accuracy and speed: high accuracy mode activates the GPS chip and waits for a satellite fix, which can take several seconds on some devices, while balanced mode uses network-assisted location for a faster response at a slightly reduced precision. The dynamic GPS buffer mechanism in the verification algorithm compensates for the lower precision by expanding the effective geofence radius in proportion to the reported accuracy value, meaning the two concerns — speed and reliability — are addressed in combination.

**bcrypt (Password Hashing).** The `bcrypt` library was used to hash user passwords before storage, with a work factor of twelve salt rounds. bcrypt was chosen over alternatives such as SHA-256 or MD5 because it is a purposefully slow hashing algorithm designed to be computationally expensive to brute-force. A work factor of twelve strikes a balance between security — producing hashes that are expensive enough to make offline dictionary attacks impractical — and performance — completing a single hash operation in a fraction of a second, acceptable for a login endpoint that handles a modest volume of authentication requests.

**JSON Web Tokens (Authentication).** JWT-based stateless authentication was chosen over server-side session management for this system. A stateless approach means the backend does not need to maintain a session store, which simplifies horizontal scaling and eliminates a shared state dependency. Each JWT encodes the user's identifier and role, enabling the backend to authorise requests without a database lookup on every request. The twenty-four-hour expiry window was chosen to provide users with a full day of uninterrupted access without requiring re-authentication during normal usage patterns, while limiting the window of opportunity if a token is compromised.

**Haversine Formula (Geofence Calculation).** The Haversine formula was implemented in a dedicated utility module in the backend and replicated in the student mobile application. The Haversine formula computes the great-circle distance between two points on a sphere given their latitude and longitude coordinates, using the Earth's mean radius of 6,371,000 metres. It was selected over a simple Euclidean distance calculation because at the scale of a university campus, the curvature of the Earth introduces a small but non-negligible error in Euclidean approximations, particularly at higher latitudes. The Haversine formula provides a geographically correct distance that is independent of the coordinate system origin, making it directly applicable to GPS coordinates.
